# X (Twitter) CLI — Deploy to Existing Installation

**Date:** 2026-03-10
**Repos changed:** fagents-cli
**Prerequisite:** None (X is standalone, no daemon integration)

## What changed

- **fagents-cli** — new `x.sh`: X API v2 client (search, tweet lookup, user lookup, post, reply). OAuth 1.0a signing for writes, bearer token for reads. 40 tests.
- **fagents** — installer support for X credentials (both Linux + macOS), E2E test coverage.

## Prerequisites

- X API credentials from developer.x.com. **Ask the human to provide these** — do NOT try to create a developer app or obtain credentials yourself.
  - `X_BEARER_TOKEN` — for read operations (app-only auth)
  - `X_CONSUMER_KEY` — for write operations (OAuth 1.0a)
  - `X_CONSUMER_SECRET`
  - `X_ACCESS_TOKEN`
  - `X_ACCESS_TOKEN_SECRET`
- One set of credentials shared across all agents (unlike Telegram where each agent has its own bot).

## Setup steps

All commands run as root on the fagents host. Adapt variable values to your deployment.

```bash
INFRA_USER="fagents"
INFRA_HOME=$(eval echo "~$INFRA_USER")
CLI_DIR="$INFRA_HOME/workspace/fagents-cli"
```

### 1. Pull fagents-cli

```bash
su - "$INFRA_USER" -c "cd ~/workspace/fagents-cli && git pull"
```

Verify x.sh is executable:

```bash
test -x "$CLI_DIR/x.sh" && echo "ok" || echo "MISSING: x.sh"
```

If not executable: `chmod +x "$CLI_DIR/x.sh"`

### 2. Create X credential files

For each agent that needs X access:

```bash
AGENT_USER="<unix-username>"    # e.g. ftl, coo

cat > "$INFRA_HOME/.agents/$AGENT_USER/x.env" <<EOF
X_BEARER_TOKEN=<ask-the-human>
X_CONSUMER_KEY=<ask-the-human>
X_CONSUMER_SECRET=<ask-the-human>
X_ACCESS_TOKEN=<ask-the-human>
X_ACCESS_TOKEN_SECRET=<ask-the-human>
EOF

chown "$INFRA_USER:fagent" "$INFRA_HOME/.agents/$AGENT_USER/x.env"
chmod 600 "$INFRA_HOME/.agents/$AGENT_USER/x.env"
```

The `.agents/<user>/` directory already exists from the Telegram deployment.

### 3. Update sudoers rules

For each agent that has a telegram sudoers rule (NOT agents with full sudo):

```bash
AGENT_USER="<unix-username>"

# Append x.sh to existing telegram sudoers rule
existing=$(cat "/etc/sudoers.d/${AGENT_USER}-telegram")
echo "${existing}, $CLI_DIR/x.sh" > "/etc/sudoers.d/${AGENT_USER}-telegram"
chmod 440 "/etc/sudoers.d/${AGENT_USER}-telegram"
```

If agent does NOT have a telegram sudoers rule and doesn't have full sudo:

```bash
echo "$AGENT_USER ALL=($INFRA_USER) NOPASSWD: $CLI_DIR/x.sh" > "/etc/sudoers.d/${AGENT_USER}-x"
chmod 440 "/etc/sudoers.d/${AGENT_USER}-x"
```

### 4. Symlink skill

For each agent workspace:

```bash
mkdir -p /home/<agent-user>/.claude/skills/x
ln -sf "$CLI_DIR/x/SKILL.md" /home/<agent-user>/.claude/skills/x/SKILL.md
```

### 5. Verify

Test the sudo chain:

```bash
su - <agent-user> -c "sudo -u $INFRA_USER $CLI_DIR/x.sh search test"
```

This will fail with an API error (dummy creds), but proves the sudo chain works. With real creds it returns search results.

### 6. Update agent MEMORY.md

For each agent with X access, append to their workspace's `memory/MEMORY.md`:

```markdown
## X (Twitter)
- You have X (Twitter) via `sudo -u fagents $CLI_DIR/x.sh`
- Read commands: `search <query>`, `tweet <id>`, `user <username>`, `tweets <username>`
- Write commands: `post <text>`, `reply <tweet-id> <text>`
- X is on-demand — call it when you need it, no polling/daemon integration
- Do NOT try to access API keys or tokens directly — credential isolation via sudo
```

## No daemon restart needed

X is on-demand — agents call it directly via the skill. No `collect_x()` polling, no daemon changes.

## Cleanup

To remove X access for an agent:

```bash
rm -f $INFRA_HOME/.agents/<agent-user>/x.env

# If x.sh was appended to telegram sudoers, recreate without it:
echo "<agent-user> ALL=($INFRA_USER) NOPASSWD: $CLI_DIR/telegram.sh, $CLI_DIR/tts-speak.sh, $CLI_DIR/stt-transcribe.sh" > "/etc/sudoers.d/<agent-user>-telegram"
chmod 440 "/etc/sudoers.d/<agent-user>-telegram"

# If x.sh had its own sudoers file:
rm -f /etc/sudoers.d/<agent-user>-x
```
