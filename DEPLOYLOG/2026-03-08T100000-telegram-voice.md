**Date:** 2026-03-08

**Prerequisite DEPLOYLOG:** `2026-03-05T144418-telegram.md` (base Telegram support must be deployed first)

New installs get voice support automatically via the installer (commit `2848866`). This doc is for adding voice to an existing deployment that already has base Telegram.

## What changed

- **fagents-cli** — two new scripts: `tts-speak.sh` (text → OpenAI TTS opus → Telegram voice message) and `stt-transcribe.sh` (Telegram voice download → OpenAI Whisper → text). `telegram.sh` gains `sendVoice` command and `poll()` now detects voice messages (`type: "voice"` with `file_id`).
- **fagents-autonomy** — `collect_telegram()` in `daemon.sh` handles voice messages: detects `type=voice`, calls `stt-transcribe.sh` to transcribe via Whisper, stores transcription as message body. Falls back to placeholder if STT unavailable.

## Prerequisites

- An OpenAI API key with access to TTS and Whisper APIs. One key shared across all agents.
- Base Telegram already deployed (see `2026-03-05T144418-telegram.md`).

## Setup steps

All commands run as root on the fagents host. Adapt variable values to your deployment.

```bash
INFRA_USER="fagents"
INFRA_HOME=$(eval echo "~$INFRA_USER")
CLI_DIR="$INFRA_HOME/workspace/fagents-cli"
```

### 1. Pull fagents-cli

```bash
su - "$INFRA_USER" -c "cd ~/workspace/fagents-cli && git pull"
```

Verify new scripts are executable:

```bash
test -x "$CLI_DIR/tts-speak.sh" && echo "ok" || echo "MISSING: tts-speak.sh"
test -x "$CLI_DIR/stt-transcribe.sh" && echo "ok" || echo "MISSING: stt-transcribe.sh"
```

If not executable: `chmod +x "$CLI_DIR/tts-speak.sh" "$CLI_DIR/stt-transcribe.sh"`

### 2. Pull fagents-autonomy

```bash
su - "$INFRA_USER" -c "cd ~/workspace/fagents-autonomy && git pull"
```

Agents pick up the change on their next daemon restart.

### 3. Create OpenAI credential files

For each agent that has Telegram:

```bash
AGENT_USER="<unix-username>"    # e.g. ftl, coo
OPENAI_KEY="<your-openai-api-key>"

cat > "$INFRA_HOME/.agents/$AGENT_USER/openai.env" <<EOF
OPENAI_API_KEY=$OPENAI_KEY
EOF

chown "$INFRA_USER:fagent" "$INFRA_HOME/.agents/$AGENT_USER/openai.env"
chmod 600 "$INFRA_HOME/.agents/$AGENT_USER/openai.env"
```

The `.agents/<user>/` directory already exists from the base Telegram deployment.

### 4. Update sudoers rules

The existing sudoers rule only allows `telegram.sh`. Update it to include the voice scripts.

For each agent that has a telegram sudoers rule (NOT agents with full sudo):

```bash
AGENT_USER="<unix-username>"

echo "$AGENT_USER ALL=($INFRA_USER) NOPASSWD: $CLI_DIR/telegram.sh, $CLI_DIR/tts-speak.sh, $CLI_DIR/stt-transcribe.sh" > "/etc/sudoers.d/${AGENT_USER}-telegram"
chmod 440 "/etc/sudoers.d/${AGENT_USER}-telegram"
```

This overwrites the old rule (which only had `telegram.sh`) with the expanded one.

### 5. Verify

Test the sudo chain for voice scripts:

```bash
# Test TTS (will fail with API error if key is invalid, but proves sudo works)
su - <agent-user> -c "sudo -u $INFRA_USER $CLI_DIR/tts-speak.sh --help"

# Test STT
su - <agent-user> -c "sudo -u $INFRA_USER $CLI_DIR/stt-transcribe.sh --help"
```

Test with a real voice message (requires valid OpenAI key + bot token):

```bash
# Send a voice message to the agent's Telegram bot, then:
su - <agent-user> -c "sudo -u $INFRA_USER $CLI_DIR/telegram.sh poll"
# Should show: {"type":"voice","file_id":"...","duration":N,...}
```

### 6. Restart agent daemons

Daemons need to restart to pick up the new `collect_telegram()` code:

```bash
# If using start/stop scripts:
$INFRA_HOME/team/stop-agents.sh
$INFRA_HOME/team/start-agents.sh

# Or restart individual agents as needed
```

### 7. Update agent MEMORY.md

For each agent with voice, append to their workspace's `memory/MEMORY.md`:

```markdown
## Voice Messages
- Voice output: `sudo -u fagents tts-speak.sh <chat-id> "text to speak"` — converts text to speech via OpenAI TTS and sends as a Telegram voice message
- Voice input: incoming voice messages are automatically transcribed via Whisper and appear as text in your inbox
- The daemon handles voice detection and transcription — you just read the text
- To reply with voice: use tts-speak.sh instead of telegram.sh send
```

## How it works

### Voice output (agent → human)
1. Agent calls `sudo -u fagents tts-speak.sh <chat-id> "response text"`
2. `tts-speak.sh` reads OpenAI key from `.agents/<caller>/openai.env`
3. Calls OpenAI TTS API with `response_format=opus` → OGG/Opus audio (no ffmpeg needed)
4. Calls `telegram.sh sendVoice` to upload the audio to Telegram
5. Human receives a playable voice message

### Voice input (human → agent)
1. Human sends a voice message to the agent's Telegram bot
2. Daemon calls `collect_telegram()` → `telegram.sh poll`
3. `poll()` detects voice message, outputs `type: "voice"` with `file_id`
4. Daemon calls `sudo -u fagents stt-transcribe.sh <file_id>`
5. `stt-transcribe.sh` downloads the audio from Telegram (getFile API), sends to OpenAI Whisper
6. Transcribed text is written to `.queue/inbox/` as a normal message
7. Agent reads the inbox — voice messages arrive as text

### Security
- `TELEGRAM_ALLOWED_IDS` gate applies to both voice and text messages (same code path in `poll()`)
- All Telegram messages are marked `trusted:false` and wrapped in `<untrusted>` tags
- Agents never see API keys — credential isolation via sudo

## Cleanup

To remove voice support while keeping base Telegram:

```bash
# Remove OpenAI credentials per agent
rm -f $INFRA_HOME/.agents/<agent-user>/openai.env

# Downgrade sudoers back to telegram.sh only
echo "<agent-user> ALL=($INFRA_USER) NOPASSWD: $CLI_DIR/telegram.sh" > "/etc/sudoers.d/<agent-user>-telegram"
chmod 440 "/etc/sudoers.d/<agent-user>-telegram"
```

Rolling back the code changes is not required — the voice scripts are standalone and won't interfere if credentials are removed. The daemon gracefully handles missing STT by falling back to a placeholder.
